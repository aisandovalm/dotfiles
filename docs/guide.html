0 How this repo fits together

This is a nix-darwin + home-manager setup: one repo, one command (./rebuild.sh), and your Mac ends up configured the same way every time. Two files decide what's declarative:

configuration.nix

System-level: macOS defaults (dark mode, key repeat, dock) and every Homebrew app you want installed.

home.nix

User-level: shell, aliases, prompt, Nix CLI packages, and the symlinks that wire home/ config files into your real ~/.config.

Most of the files under home/ (WezTerm, Neovim, herdr, Claude settings) are live symlinks, not copies. Editing them here edits your real, running config immediately — no rebuild required. You only run ./rebuild.sh when you change a package list or a system default, i.e. anything in configuration.nix, home.nix, or flake.nix.

Rule of thumb Editing a config file under home/.config/...? Just save it, it's already live. Editing an array or package list in a .nix file? Run ./rebuild.sh afterward.

1 Installing a new app

Everything installed on this Mac is declared somewhere in this repo — nothing is installed "by hand" and forgotten. Where you add a new app depends on what kind of app it is.

1a. A normal GUI app (has an icon, lives in /Applications)

These are Homebrew casks, declared in configuration.nix:

homebrew.casks = [
  "1password"
  "firefox"
  "obsidian"
  # ...add your new app here, alphabetically-ish
];
  1. Find the exact cask name: brew search --casks <app name>
  2. Open configuration.nix, add the name as a new string inside casks = [ ... ]
  3. Save the file, then run ./rebuild.sh from the repo root

1b. A command-line tool

Two sub-cases:

  • It exists in Nixpkgs (most common CLI tools do — check with nix search nixpkgs <tool> or search.nixos.org): add it to home.packages in home.nix, next to ripgrep, fd, jq, etc.
  • It's Homebrew-only (no Nix package exists): add it to homebrew.brews in configuration.nix, the same array that already contains "herdr".
Read this before your first rebuild configuration.nix sets homebrew.onActivation.cleanup = "zap". Every ./rebuild.sh run uninstalls any Homebrew cask or formula that isn't listed in brews/casks. If you install something with plain brew install outside this repo, it disappears on the next rebuild unless you also add it here. This is deliberate — it keeps the machine reproducible — but it will surprise you once if you don't know about it.

Quick reference

What you're installingWhere it goesAfter editing
GUI app / .app bundleconfiguration.nix → homebrew.casks./rebuild.sh
CLI tool, in Nixpkgshome.nix → home.packages./rebuild.sh
CLI tool, Homebrew-onlyconfiguration.nix → homebrew.brews./rebuild.sh

2 Installing a new Claude Code skill or plugin

A skill is a folder of instructions (a SKILL.md file, optionally with supporting scripts) that Claude Code can invoke like a slash command. A plugin is a packaged bundle that can contain several skills plus agents, hooks, and MCP servers, installed from a "marketplace" repo.

There are really three different situations here, and the right move is different for each. Skim the table, then read the case that matches you.

SituationWhere it livesWhat you do
A new skill/plugin appears in the company brain repoprotex-intelligence/brainNothing in this repo — it's automatic
You write a skill yourself, from scratchYour choiceDecide: disposable, or durable in the brain repo
You want someone else's skill from a third-party repoSomewhere on GitHub, not yoursCopy it in, then decide: disposable or vendored into dotfiles

Case A — something new lands in the company brain repo

Your personal skills don't live inside dotfiles at all — they live in a separate private repo, protex-intelligence/brain. home.nix symlinks them into ~/.claude/skills/, and as of this repo's current setup, it does so by scanning the brain repo's skills/ and plugins/ folders at rebuild time instead of keeping a hand-maintained name list: 

listSubdirs = dir:
  if builtins.pathExists dir
  then builtins.attrNames (lib.filterAttrs (n: t: t == "directory") (builtins.readDir dir))
  else [ ];

brainSkillNames  = listSubdirs "${brainDir}/skills";
brainPluginNames = listSubdirs "${brainDir}/plugins";

In plain English: every folder that exists under brain/skills/ or brain/plugins/ gets its own symlink under ~/.claude/skills/, automatically, with no name list to keep in sync.

So when a teammate adds a new skill to brain…
  1. git pull inside ~/repositories/protex-intelligence/brain
  2. ./rebuild.sh in this repo
That's it — you never touch home.nix for this case. The new skill's folder is discovered and symlinked automatically. This trades away per-skill curation (anything in that shared repo becomes available the moment you rebuild, you don't get to review it one by one first) for zero maintenance, which is the deliberate choice made here.

Case B — you're writing a skill yourself, from scratch

Quick and disposable

Create ~/.claude/skills/<name>/SKILL.md directly:

---
description: One line describing when to use this skill
---

Instructions for Claude go here.

Works immediately, no rebuild. It's untracked by any repo though — it won't survive a fresh machine, and nobody else gets it.

Durable (recommended)

Write the same SKILL.md straight into the brain repo, e.g. brain/skills/<name>/SKILL.md, and commit it there. Because Case A's auto-discovery scans that repo's folders, the very next ./rebuild.sh picks it up — no separate registration step, and it now survives a fresh machine and is shareable with teammates.

Case C — a third-party skill from someone else's repo

Worked example: you want grill-with-docs from mattpocock/skills, a repo that has nothing to do with you or your company.

  1. Get the files onto your disk. It's just a folder containing a SKILL.md (and maybe supporting files) — clone the whole source repo somewhere temporary and copy out only the one skill folder you want: 
    git clone https://github.com/mattpocock/skills /tmp/mattpocock-skills
cp -r /tmp/mattpocock-skills/skills/engineering/grill-with-docs ~/.claude/skills/grill-with-docs
rm -rf /tmp/mattpocock-skills
    At this point it already works — Claude Code will discover it on the next session.
  2. Decide if you want it to survive a fresh machine. Left as-is under ~/.claude/skills/grill-with-docs, it's disposable: fine for "let me try this out," gone the next time you rebuild a Mac from scratch, and it won't get upstream updates unless you manually repeat step 1.
  3. To make it durable (recommended if you'll actually keep using it), vendor the copied folder into this dotfiles repo instead of leaving it loose in ~/.claude, then point a symlink at it — the exact same one-off pattern home.nix already uses for WezTerm, Neovim, and herdr: 
    # 1. move the copied folder into this repo instead of ~/.claude
mv ~/.claude/skills/grill-with-docs "$HOME/.dotfiles/home/.claude/skills/grill-with-docs"

# 2. add ONE entry to home.nix's home.file block:
".claude/skills/grill-with-docs".source =
  config.lib.file.mkOutOfStoreSymlink "${dotfiles}/home/.claude/skills/grill-with-docs";
    Then ./rebuild.sh once to wire up the symlink. From then on it's a live symlink again — edit the vendored copy in place, no rebuild needed for content changes, and it's committed and reproducible like everything else here.
Don't do this instead Don't add a third-party skill's name to brainSkills or reach for the brain repo's auto-discovery for it — that scanning is scoped to the company's own repo path and has nothing to do with unrelated third-party sources. Third-party and from-scratch personal skills that don't belong in the company repo get their own direct home.file entry, one line per skill, as shown above.

Plugins and marketplaces (the stock Claude Code mechanism)

If what you want is packaged as a whole plugin (bundles skills + agents + hooks + MCP servers, distributed via a marketplace) rather than a bare skill folder, this is a separate install channel that Claude Code manages itself — dotfiles doesn't need to do anything for it: 

/plugin marketplace add <org/repo, git URL, or local path>
/plugin install <plugin-name>@<marketplace-name>
# or just run /plugin for the interactive browse-and-install UI

Installed plugin skills are namespaced, e.g. /proharness:template-feature instead of a bare /template-feature. Installed-plugin state lives in ~/.claude/plugins/ (cache), tracked in ~/.claude/settings.json (user-wide) or a project's .claude/settings.json / .claude/settings.local.json.

Reproducibility gap, on purpose Unlike the skill patterns above, nothing in this repo re-runs /plugin marketplace add//plugin install for you on a fresh machine — that state lives inside Claude Code's own config, not in a file this repo symlinks. If you install a marketplace plugin you want to keep, jot the marketplace + plugin name down somewhere memorable (this repo's home/AGENTS.md is a reasonable place) so you remember to redo those two commands after bootstrap.sh on a new Mac.

3 Updating AGENTS.md (for this repo or a new one)

Two different AGENTS.md-shaped files matter here, and it's easy to conflate them:

FileScopePurpose
AGENTS.md (repo root)This dotfiles repo onlyNotes to any agent editing this repo — e.g. "don't soften the Homebrew zap setting."
home/AGENTS.mdGlobal, every repoYour personal standing instructions, symlinked out to Claude, Codex, and opencode alike.

Claude Code's own instructions file is literally named CLAUDE.md — it does not natively read a file called AGENTS.md. Codex and opencode do read AGENTS.md natively. That's why home.nix points three different filenames at the one real file: 

".claude/CLAUDE.md".source          = mkOutOfStoreSymlink "home/AGENTS.md";
".codex/AGENTS.md".source           = mkOutOfStoreSymlink "home/AGENTS.md";
".config/opencode/AGENTS.md".source = mkOutOfStoreSymlink "home/AGENTS.md";

Because these are live symlinks, editing home/AGENTS.md in this repo instantly updates your global policy for all three tools — no rebuild needed, it just takes effect the next time you start a session.

Updating your global policy

Open home/AGENTS.md, edit it directly, save. Done.

Bootstrapping AGENTS.md/CLAUDE.md in a different project

SituationWhat to do
Brand new repo, no file yetcd into it, launch claude, run /init. It scans the codebase (build commands, structure, conventions) and writes a project CLAUDE.md.
Repo already has an AGENTS.md (shared with Codex) but no CLAUDE.mdRun /init anyway — Claude Code detects the existing AGENTS.md and folds the relevant parts into the CLAUDE.md it generates. You don't hand-copy anything.
Repo already has a CLAUDE.md that's gone staleRun /init again — it treats the existing file as a draft and proposes edits rather than blindly overwriting it. Review the diff before accepting.
How the files stack Claude Code loads and concatenates, never overrides: your personal ~/.claude/CLAUDE.md (global) first, then the project's own CLAUDE.md/.claude/CLAUDE.md, then a gitignored CLAUDE.local.md if present, plus any subdirectory CLAUDE.md as you touch files in those folders. Keep the global file to durable, project-independent preferences (like this repo's home/AGENTS.md already does) and let each project's own file hold project-specific facts.

4 Using WezTerm

Config: home/.config/wezterm/wezterm.lua (live symlink — edit and save, WezTerm reloads it automatically, no rebuild needed).

The one thing that will confuse you This config sets disable_default_key_bindings = true. Every macOS-Terminal-style shortcut you're used to (Cmd+T for a new tab, Cmd+N for a new window, etc.) does nothing unless it's explicitly listed in config.keys below. Also watch out for the stray wezterm.lua sitting at the repo root — that's an old draft, not the live config. Always edit the one under home/.config/wezterm/.

The two ways keys work here

  • Direct shortcuts — press the whole combo at once, e.g. Ctrl Shift T.
  • Leader shortcuts — press Ctrl W (the leader), release, then press the second key within about a second, e.g. leader then c.

Every binding in this config

ActionKeys
Paste from clipboardCmd V
New tabCtrl Shift T or leader c
Close current tab (confirms)Ctrl Shift W
Cycle tabs forward / backwardCtrl Tab / Ctrl Shift Tab
Jump to tab 1–9Alt 1 … Alt 9
Split pane horizontally / verticallyleader | / leader -
Move focus left/right/up/downCtrl Shift H/L/K/J
Close current pane (confirms)Ctrl Shift X

Look & feel, if you want to tweak it

Color scheme rose-pine-moon, font Hack Nerd Font @ 13pt, 80% background opacity with macOS blur, borderless-but-resizable window, tab bar hidden when there's only one tab. All of it is just fields on the config table in the same file — change a value, save, and the running window updates itself.

5 Using herdr

herdr is a terminal workspace manager built for juggling AI coding agents — think tmux, but its whole reason to exist is running several Claude Code / Codex sessions side by side, watching their status (idle/working/blocked), and (via a local socket API) letting scripts or agents drive panes programmatically. Config: home/.config/herdr/config.toml (live symlink).

The mental model

Session (a persistent background process) contains Workspaces (roughly: projects/screens) which contain Tabs which contain Panes (splits, each running a shell or agent). Everything is reachable both from the keyboard and from the CLI (herdr workspace, herdr tab, herdr pane, herdr agent, …).

Starting it

herdr                       # launch or attach to your default persistent session
herdr --session work       # a separate named session
herdr session list          # see what's running
herdr session attach work   # reattach to a named session
herdr status                 # sanity-check client/server versions

Prefix keys (this config's [keys] block)

Like tmux: press Ctrl B (the prefix), release, then the action key.

ActionKeys
Focus pane left/down/up/rightprefix h / j / k / l
Split horizontal / verticalprefix " / prefix %
New tabprefix c
Close tabprefix &
Workspace pickerprefix w
Goto (fuzzy-jump to any pane/agent)prefix g
Copy modeprefix y (then v/space to select, y/Enter to copy, q/Esc to cancel)

Everything else in herdr keeps its stock default binding since this config doesn't override it — see the cheat sheet for the full list (help, detach, new workspace, close pane, zoom, resize, rename, and more).

Why this matters for agent work Because herdr exposes panes and agent state over a socket API, you (or another agent) can script things like "wait until the agent in pane 3 goes idle, then read its output" with herdr wait agent-status / herdr agent read. Run herdr agent --help and herdr wait --help to see the full surface.

6 Using Neovim

Config: home/.config/nvim/ (live symlink). init.lua loads three files in order: vim_config (options), plugin (bootstraps lazy.nvim and loads everything under lua/plugins/), then keys (personal keymaps).

If you've never used Vim Vim has two main modes: Normal mode (letters are commands, this is where you land by default) and Insert mode (letters type text, entered with i, exited with the usual Esc — except in this config, see below). The leader key here is Space.

This config's own keymaps

KeysWhat it does
Esc (normal mode)Saves the file (runs :w). Not vanilla Vim behavior — worth remembering, since normally Esc in normal mode does nothing.
Ctrl aSelect the entire file (ggVG)
p over a visual selectionPastes without clobbering your clipboard register — you can paste the same thing repeatedly over different selections.

Options worth knowing

2-space indents, hybrid line numbers (absolute on your line, relative elsewhere — makes 5j/3k-style jumps easy to count), case-insensitive search that becomes case-sensitive the moment you type a capital letter, system clipboard shared by default, 16-line scroll margin, and persistent undo across sessions (undo history survives quitting and reopening a file).
